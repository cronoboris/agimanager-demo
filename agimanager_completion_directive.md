# AGI Manager 완성도 향상 총괄 지침서

## 문서 목적
이 문서는 **AGI Manager의 현재 구현물을 바탕으로**, 게임을 단순히 "동작하는 전략 시뮬레이션"에서 **깊이 있는 대전략형 경영 시뮬레이션**으로 끌어올리기 위해 필요한 추가 개발과 폴리싱 항목을 **체계적으로 정리한 실행 지침서**다.

이 문서를 읽는 AI의 목표는 다음과 같다.

1. 현재 구현된 시스템의 범위와 한계를 정확히 이해한다.
2. 시스템별 결핍이 무엇인지 파악한다.
3. 최소 구현이 아니라, **완성도와 반복 플레이성, 기억성, 설명 가능성**을 높이는 방향으로 설계를 확장한다.
4. 구현 순서를 스스로 계획하되, 이 문서의 **의존성**, **품질 기준**, **금지 사항**을 따른다.

이 문서는 일정표가 아니다. 월 단위 로드맵 대신, **무엇을 반드시 더 만들어야 하고**, **무엇을 얼마나 깊게 다듬어야 하며**, **어떤 순서로 연결해야 하는지**를 정의한다.

---

## 현재 구현 스냅샷
다음은 실제 패키지 기준 관찰된 현재 상태다.

- 코드 규모: `js/main.js` 6395줄, `EconomySystem.js` 1277줄, `ComputeSystem.js` 1111줄, `ChipProgramSystem.js` 853줄, `WebGLMap.js` 824줄
- 데이터 규모:
  - 기술 91개
  - 국가 118개, 이 중 시작 국가로 플레이 가능한 국가는 12개
  - 경쟁사 6개
  - GPU 카탈로그 11개
  - 모델 아키텍처 8개
  - 파라미터 규모 7단계
  - 훈련 단계 5개
  - 배포 전략 5개
  - 벤치마크 6개
  - 역사 이벤트 83개
  - 랜덤 이벤트 96개
  - 기술 이정표 이벤트 16개
  - 선택지 보유 이벤트 119개
- 시스템 현황:
  - 연구, 인재, 경제, 투자, 컴퓨트, GPU 구매/배치, 데이터, 모델 설계/훈련/배포, 서비스 채널, 내부 AI 활용, 내부 칩 프로그램, 업적, 통계, 세이브/로드, 다국어, 2D/3D 지도, 튜토리얼, 사운드, 레거시 보너스가 이미 존재
- 안정성 현황:
  - 테스트 28개 모두 통과
  - 저장 마이그레이션, 이벤트 체인, 밸런스, 모델 서비스, 튜토리얼, 지도, 국가 modifier, 내부 AI, 칩 프로그램 관련 회귀 테스트 존재
- 기술 부채 신호:
  - `window.game` 직접 참조 약 65회
  - `main.js` 집중도가 매우 높음
  - `dist-native/`가 소스의 일부 신규 모듈을 반영하지 않는 정적 복제본 위험 존재
  - `build-demo.mjs`에서 이벤트 트리밍이 로그만 남기고 실제 동작하지 않는 구간 존재
  - UI 버전 문자열 일부가 아직 `v0.2.0`으로 남아 있음
  - 문서상의 수치와 실구현 수치가 여러 곳에서 어긋나기 시작함

이 스냅샷의 의미는 명확하다. **기초 시스템은 넓게 깔려 있고, 게임은 이미 플레이 가능한 구조를 갖췄다. 그러나 세계의 행위성, 중후반 구조, 설명 가능성, 장기 반응성, 인물 서사, 기술 구조 정리가 아직 부족하다.**

## 최상위 방향성
앞으로의 개발은 다음 여섯 가지 원칙을 따라야 한다.

### 1. 폭보다 밀도를 우선한다
이미 시스템 종류는 충분하다. 이제 필요한 것은 더 많은 메뉴가 아니라, **서로의 결과를 바꾸는 상호작용**이다.

### 2. 선택은 숫자 변화가 아니라 질서 변화가 되어야 한다
좋은 이벤트와 시스템 선택은 `+10%`가 아니라, 이후 12개월에서 5년까지 규칙을 바꿔야 한다.

### 3. 경쟁 상대는 수치 곡선이 아니라 의도를 가진 세력이어야 한다
경쟁사는 단순 성장 공식이 아니라 **교리, 기억, 목표, 정치적 위치, 자원 압박**을 가져야 한다.

### 4. 후반부는 확장판이 아니라 본편이어야 한다
게임의 가장 재밌는 구간이 초반 생존 이후 열리도록 재설계해야 한다. 현재는 중후반 콘텐츠보다 승패 판단이 먼저 올 위험이 있다.

### 5. 플레이어는 항상 "왜 이런 결과가 나왔는가"를 이해할 수 있어야 한다
복잡성은 허용되지만 불투명성은 허용되지 않는다. 보고서, 로그, 툴팁, 원인 분석 화면이 필요하다.

### 6. 최소구현을 피한다
각 시스템은 "보였다" 수준이 아니라, **반복 플레이에서 다르게 읽히고**, **상호작용을 낳고**, **기억에 남는 결과를 생성하는 수준**까지 확장해야 한다.

---

## AI가 이 문서를 읽고 계획을 짤 때의 운영 규칙

1. 먼저 **블로커**, **심화 시스템**, **폴리싱**, **콘텐츠 확장**을 구분한다.
2. 기능을 추가하기 전에 반드시 아래를 확인한다.
   - 저장 호환성
   - 기존 이벤트/시스템과의 연결 지점
   - UI 설명 가능성
   - 테스트 추가 필요성
3. 새 기능은 가능하면 데이터 주도형으로 설계한다. 하드코딩 분기문을 늘리는 방식은 지양한다.
4. 기존 시스템을 우회하는 새 미니게임을 만들지 않는다. 연구, 경제, 이벤트, 지도, 인재, 모델, 외교 루프와 반드시 연결한다.
5. UI 구현 전에 게임 상태 모델과 규칙을 먼저 안정화한다.
6. 신규 시스템은 항상 다음 네 가지를 함께 설계한다.
   - 상태 구조
   - 월간/일간 처리 규칙
   - UI 표면
   - 회귀 테스트
7. 텍스트 콘텐츠를 늘릴 때는 i18n 키와 fallback 체계를 같이 넣는다.
8. 이벤트나 엔딩을 추가할 때는 반드시 **후속 상태**, **재등장 조건**, **다른 시스템에 미치는 파장**을 정의한다.

# 1. 즉시 해결해야 하는 블로커

## 1.1 문서와 구현의 진실 원천을 단일화
### 문제
현재 개발 문서의 수치와 실제 구현 수치가 여러 곳에서 어긋난다. 이 상태가 지속되면, 이후 AI가 문서를 보고 작업할 때 잘못된 기준으로 확장하게 된다.

### 해야 할 일
- `docs/` 내 시스템 설명 문서를 실제 데이터 파일 기준으로 자동 갱신 가능한 구조로 재편한다.
- 데이터 카운트, 아키텍처 수, 이벤트 수, 플레이 가능 국가 수 등은 수동 기재를 줄이고 스크립트로 생성한다.
- `docs/DEVELOPMENT_REPORT.md`는 히스토리 문서로 두고, 별도의 `CURRENT_STATE.md`를 만들어 현재 진실 원천으로 지정한다.
- 코드와 문서가 어긋나는 영역을 표로 정리한다.
  - 기술 수
  - 아키텍처 수
  - 이벤트 수
  - 플레이 가능 국가 수
  - 승리 조건 수
  - 주요 UI 탭 구조

### 완료 기준
- AI가 문서를 읽었을 때 실제 코드와 어긋나는 핵심 수치가 없어야 한다.
- 시스템별 현재 상태가 자동 검증 또는 자동 생성 스크립트로 갱신 가능해야 한다.

---

## 1.2 배포 산출물 경로 정리
### 문제
`dist-native/`가 소스 트리의 일부 최신 모듈을 반영하지 않는 중복 복제본이 될 위험이 있다. 이런 복제 구조는 버그 재현과 빌드 신뢰성을 해친다.

### 해야 할 일
- `dist-native/`는 커밋된 정적 산출물이 아니라 빌드 결과물로만 취급하도록 정리한다.
- 소스와 산출물을 동시에 수정하는 흐름을 제거한다.
- 빌드 스크립트가 항상 최신 `js/`, `css/`, `data/`, `locales/`를 복제하는지 검증 테스트를 추가한다.
- 누락 모듈 검사를 CI에 넣는다.

### 완료 기준
- 실행 가능한 네이티브 번들이 항상 최신 소스와 동일한 기능 집합을 포함해야 한다.
- `dist-native/`가 stale 상태가 되어도 CI에서 탐지되어야 한다.

---

## 1.3 데모 빌드 신뢰성 확보
### 문제
`build-demo.mjs`에는 이벤트 필터링이 실제로 적용되지 않는 구간이 있다. 데모와 정식 버전의 차이를 보장하지 못하면 QA와 배포 품질이 낮아진다.

### 해야 할 일
- 데모 빌드의 잘라내기 규칙을 데이터/소스 양쪽에서 실제 적용되도록 수정한다.
- 데모에서 비활성화된 시스템과 숨겨진 시스템을 명시적으로 관리한다.
- 데모 빌드 전용 검증 스크립트를 만든다.
  - 최대 연도 제한
  - 사용 가능 노선 제한
  - 경쟁사 수 제한
  - 기술 트리 제한
  - 이벤트 제한
- 데모에서 도달 가능한 엔딩과 불가능한 엔딩을 명시하고 UI에 반영한다.

### 완료 기준
- 데모 빌드 결과물만 보고도 정식 버전 기능 일부가 확실히 제거되었음을 검증할 수 있어야 한다.

---

## 1.4 버전, 빌드 메타데이터, 저장 호환성 관리 강화
### 문제
UI 표기 버전과 문서 버전, 실제 빌드 버전이 섞여 있다. 기능이 늘수록 저장 마이그레이션 충돌 위험도 커진다.

### 해야 할 일
- 버전 문자열의 단일 원천 파일을 만든다.
- 메뉴, 크레딧, 저장 파일 메타데이터, 릴리즈 로그가 모두 같은 버전 정보를 사용하게 한다.
- 저장 버전 마이그레이션 테이블을 명문화한다.
- 세이브 파일에 다음을 추가한다.
  - 빌드 해시 또는 콘텐츠 해시
  - 활성 시스템 플래그
  - 데이터 버전
  - 모드/데모 여부

### 완료 기준
- 저장 파일 호환성이 깨질 수 있는 변경은 모두 마이그레이션 단계를 거치게 되어야 한다.

# 2. 캠페인 구조와 게임 페이싱 재설계

## 2.1 캠페인을 4막 구조로 재편
### 현재 한계
현재 게임은 2017부터 2031까지 이어지는 단일 선형 러닝에 가깝다. 초반 생존 압박은 있으나, 시대가 바뀌며 질문 자체가 바뀌는 느낌은 약하다.

### 목표
게임을 다음 4막 구조로 재편한다.

1. **창업기**
   - 생존, 첫 채용, 첫 연구, 첫 모델, 첫 투자
2. **확장기**
   - 조직 분화, 제품-연구 균형, 클라우드 의존 vs 자체 인프라
3. **정치경제기**
   - 규제, 수출 통제, 국가 정렬, 조달, 로비, 공급망
4. **프런티어기**
   - AGI 안전성, 국제 거버넌스, 이사회 충돌, 내부 분열, 세계 질서 재편

### 구현 항목
- 각 막마다 해금되는 UI 탭이 아니라, **의사결정 구조 자체**를 늘린다.
- 막 전환 조건을 연도만으로 두지 않고 다음 요소 조합으로 정의한다.
  - 팀 규모
  - 최고 모델 성능
  - 투자 단계
  - 글로벌 AI 수준
  - 규제 강도
  - 경쟁사 행동
- 각 막 전환 시 다음 중 최소 둘 이상이 바뀌게 한다.
  - 월간 처리 로직
  - 랜덤 이벤트 풀
  - 투자자 요구사항
  - 경쟁사 전략
  - 국가별 리스크
  - 엔딩 접근성
- 막 전환 연출을 넣는다.
  - 업계 리포트
  - 뉴스 몽타주
  - 시장 기대치 변화
  - UI 색조 및 사운드 변화

### 완료 기준
- 같은 플레이어라도 막이 바뀌면 "같은 숫자를 더 크게 굴리는 중"이 아니라 "이제 다른 질문에 답해야 한다"는 감각이 들어야 한다.

---

## 2.2 중반부 체류 시간을 늘리고 후반부를 실질화
### 현재 한계
현재 경쟁사 성장과 AGI 임계값 구조상, 중후반 콘텐츠가 열리기 전에 판이 종료될 위험이 있다.

### 해야 할 일
- 경쟁사 AGI 도달 속도를 재조정한다.
- 플레이어와 경쟁사의 성장曲선을 분리한다.
  - 초반: 플레이어가 생존과 초기 선택으로 격차를 줄이는 구간
  - 중반: 플레이어의 전략 차별화가 본격 반영되는 구간
  - 후반: 세계 질서와 안전성, 정치가 기술 못지않게 중요해지는 구간
- 후반부 전용 리소스 병목을 만든다.
  - 전력
  - 규제 승인
  - 고급 인재 유지
  - 데이터 합법성
  - 국제 신뢰도
  - 사회적 정당성
- "AGI 직전"을 별도 단계로 만든다.
  - 단순 성능 100이 아니라, 능력 축 조합과 제도적 승인, 안전성 검증, 사회적 수용성 중 일부를 요구한다.

### 완료 기준
- 1회 플레이에서 2024년 이후 콘텐츠가 자주 실질적으로 사용되어야 한다.
- 후반부가 단순 승리 판정 버튼이 아니라 가장 복잡하고 흥미로운 구간이 되어야 한다.

# 3. 경쟁사 AI와 외교의 전면 확장

## 3.1 경쟁사를 "교리를 가진 행위자"로 재설계
### 현재 한계
지금 경쟁사는 연구 성장, 모델 출시, 시장점유율 요동, 관계 드리프트 수준에 머무는 경향이 강하다.

### 목표
각 경쟁사는 단순 수치 묶음이 아니라 다음을 가진다.
- 장기 목표
- 위험 선호도
- 자금 성향
- 배포 성향
- 정치적 후원
- 기술 교리
- 플레이어와의 기억

### 구현 항목
경쟁사마다 최소 아래 항목을 추가한다.

1. **전략 교리**
   - Frontier Rush
   - Safety First
   - Platform Lock-in
   - Government Contractor
   - Open Ecosystem
   - Hardware Vertical Integration

2. **행동 예산**
   - 연구 투자
   - PR/마케팅
   - 로비
   - 인재 스카우트
   - 가격 인하
   - 파트너십
   - 인수합병

3. **장기 목표**
   - 세계 최고 모델
   - 정부 계약 독점
   - 오픈소스 지배
   - 반도체 자립
   - 특정 지역 패권

4. **기억 시스템**
   - 플레이어가 스카우트했는지
   - 공동연구를 깼는지
   - 규제기관에 찔렀는지
   - 가격전쟁을 시작했는지
   - 기자와 여론전에서 공격했는지

### 폴리싱 항목
- 경쟁사별 발표문, 언론 톤, 시각 브랜딩을 차별화한다.
- AI 리더보드가 단순 성능표가 아니라 "지금 무슨 전략을 쓰는가"를 보여주도록 바꾼다.

### 완료 기준
- 같은 경쟁사라도 판마다 다른 교리와 상황에 따라 다른 선택을 하게 되어야 한다.
- 플레이어가 특정 경쟁사를 "저 회사는 늘 이 패턴으로 압박한다"고 기억할 수 있어야 한다.

---

## 3.2 외교를 거래 메뉴에서 정치 시스템으로 확장
### 현재 한계
관계 개선, 공동 연구, 인재 스카우트, M&A, 라이선싱, 소송 등 표면은 있으나, 외교가 장기 판도나 블록 형성으로 이어지는 정도는 아직 약하다.

### 구현 항목
- 경쟁사 외교를 **조약/관계 상태** 기반으로 확장한다.
  - 연구 동맹
  - API 상호 공급
  - 인재 비밀협약
  - 비방전 중
  - 소송전 중
  - 정부 조달 경쟁 금지 협약
  - 데이터 공유 협정
- 외교 상태는 기간, 파기 비용, 숨은 불만, 신뢰도 변수를 가진다.
- 외교 결과가 다음에 반영되어야 한다.
  - 투자자 신뢰
  - 정부 계약 접근성
  - 인재 시장 호감
  - 규제기관 시선
  - 오픈소스 생태계 반응
- 경쟁사가 플레이어에게 제안하거나 협박하도록 만든다.

### 완료 기준
- 외교가 단기 수치 거래가 아니라 세력 균형과 서사 생성의 중심축이 되어야 한다.

---

## 3.3 경쟁사 내부 상태 시뮬레이션 추가
### 구현 항목
- 각 경쟁사에 다음 내부 상태를 둔다.
  - 현금 압박
  - 연구병목
  - 안전 스캔들 누적치
  - 규제 리스크
  - 인재 유출 위험
  - GPU 의존도
  - 정부 의존도
- 플레이어가 정보기관, 언론, 업계 인맥, 분석 보고서를 통해 이 상태를 추정할 수 있게 한다.
- 정보가 불완전하도록 구성한다.
  - 확정 정보
  - 루머
  - 추정치
  - 오보

### 완료 기준
- 경쟁사를 상대할 때 완전 정보 퍼즐이 아니라 제한 정보 하 전략 판단이 되도록 한다.

---

# 4. 국가, 지정학, 규제 시스템의 체급 상승

## 4.1 국가를 modifier 집합에서 정책 행위자로 승격
### 현재 한계
국가별 보너스는 많이 연결되었지만, 국가는 아직 능동적으로 정책을 바꾸는 행위자라기보다 강한 modifier 카드에 가깝다.

### 목표
국가가 시장, 인재, 규제, 에너지, 군사, 데이터, 반도체에서 능동적으로 움직이게 만든다.

### 구현 항목
- 각 국가에 다음 정책 축을 둔다.
  - 데이터 개방도
  - 개인정보 규제
  - 수출통제 수준
  - 반독점 강도
  - 국방 AI 우선도
  - 친기업성
  - 에너지 안정성
  - 외국인 인재 유입 허용도
  - 탄소/환경 규제
  - 자국기업 선호도
- 국가는 연도, 글로벌 AI 수준, 실업률, 지정학적 긴장, 플레이어 행동에 따라 정책을 조정한다.
- 플레이어는 국가에 대해 다음 행동을 할 수 있다.
  - 본사 이전
  - 리전 오피스 설립
  - 데이터센터 건설
  - 현지 로비
  - 대학 제휴
  - 조달 입찰
  - 규제 대응 로드맵 제출
  - 현지화 전략 선택

### 완료 기준
- "어느 국가에 있느냐"가 초반 보너스 이상으로, 회사의 장기 전략과 위험 구조를 바꾸어야 한다.

---

## 4.2 지역별 시장과 규제를 분리
### 현재 한계
시장과 규제가 글로벌 합산 감각에 가까워, 지역 전략의 손맛이 충분히 살아나지 않는다.

### 구현 항목
- 시장을 최소 다음 단위로 분리한다.
  - 북미
  - 유럽
  - 중국
  - 동북아
  - 동남아
  - 중동
  - 인도권
  - 아프리카
  - 남미
- 각 지역은 다음을 가진다.
  - TAM/SAM 성격
  - 주력 수요 채널
  - 규제 리스크
  - 현지 경쟁 강도
  - 배포 선호
  - 가격 민감도
  - 신뢰 민감도
- 배포 전략과 지역성이 결합하게 한다.
  - 소비자 챗은 문화/언어/규제 민감
  - 엔터프라이즈는 산업 기반과 통합 비용 민감
  - 정부 계약은 외교 관계, 안보 블록, 신뢰성 민감

### 완료 기준
- 같은 모델이라도 지역마다 전혀 다른 성과와 리스크를 내야 한다.

---

## 4.3 지정학을 월간 수치에서 국제 사건 시스템으로 확장
### 현재 한계
지정학적 긴장 수치는 존재하지만, 국제 관계의 동적 재편과 플레이어 전략 변경을 촉발하는 사건축이 충분히 강하지 않다.

### 구현 항목
- 지정학 주요 사건군을 만든다.
  - 반도체 제재
  - 국방 AI 연합
  - 데이터 현지화 블록
  - 국제 안전 협약
  - 에너지 가격 급등
  - 해저 케이블/클라우드 분쟁
  - 대규모 사이버 공격
- 이 사건은 단발성 팝업이 아니라 **지속 상태**를 만든다.
- 국가 블록 개념을 추가한다.
  - 미국권
  - 중국권
  - EU 거버넌스 블록
  - 비동맹/기회주의 블록
- 플레이어가 어느 블록에 가까운지에 따라 얻는 것과 잃는 것이 달라진다.

### 완료 기준
- 후반부의 전략 판단이 기술 트리만 보지 않고 국제 정렬을 고려하도록 바뀌어야 한다.

---

# 5. 조직, 인재, 이사회, 투자자 드라마 확장

## 5.1 인재를 스탯 카드에서 관계 기반 인물로 확장
### 현재 한계
인재는 능력치, 특성, 충성도, 이념 등을 가지지만, 장기적인 서사와 관계망이 강하지 않다.

### 구현 항목
각 인재에 최소 다음을 추가한다.
- 경력사와 연구 분야 서사
- 업계 명성
- 목표 성향
  - 논문지향
  - 제품지향
  - 안전지향
  - 부유해지기
  - 사회변혁
  - 국가충성
- 인간 관계
  - 친구
  - 멘토/제자
  - 경쟁관계
  - 창업 멤버
  - 투자자와 연결
- 갈등 트리거
  - 저임금
  - 과로
  - 안전 우려
  - 오픈소스 배신
  - 군사 계약 반발
  - 인종/국적 차별 이슈
  - 연구 방향 불만

### 폴리싱 항목
- 주요 인재는 반복 이벤트에 재등장하게 한다.
- 퇴사는 단순 사라짐이 아니라 경쟁사 합류, 창업, 내부고발, 기자 제보, 대학 복귀 등 다양한 결과를 갖게 한다.

### 완료 기준
- 플레이어가 특정 인재를 능력치가 아니라 이름과 스토리로 기억할 수 있어야 한다.

---

## 5.2 팀 구조와 조직 운영 도입
### 현재 한계
인재 배치가 주로 연구와 훈련 단위 중심이라, 회사가 커졌을 때 조직 관리의 감각이 약하다.

### 구현 항목
- 팀/부서 체계를 도입한다.
  - Frontier Research
  - Product
  - Applied Research
  - Infra
  - Safety
  - Policy/Legal
  - People/Recruiting
  - Government Relations
- 부서별 리더를 지정할 수 있게 한다.
- 부서별 KPI와 마찰을 둔다.
- 인원이 커지면 관리비와 커뮤니케이션 페널티가 생긴다.
- 부서 분화가 투자, 속도, 품질, 사고율에 영향을 준다.

### 완료 기준
- 회사 규모가 커질수록 "사람을 더 뽑으면 무조건 좋다"가 아니라 조직 구조 최적화 문제가 생겨야 한다.

---

## 5.3 이사회와 투자자를 독립 세력으로 구현
### 현재 한계
투자는 주로 라운드 진행과 수치 충족에 가깝고, 투자자 요구와 거버넌스 갈등이 약하다.

### 구현 항목
- 투자자 archetype 추가
  - 초고속 성장형 VC
  - 안전 중시 재단
  - 국부펀드
  - 전략적 기업 투자자
  - 국방/정부 자금
- 투자 라운드 후 이사회 자리를 부여한다.
- 이사회가 안건을 올리게 한다.
  - CEO 교체 압박
  - 연구 방향 변경
  - 구조조정
  - 정부 계약 확대
  - 매각/합병
  - IPO 강행
- 안건 무시 시 신뢰와 자금 조달, 언론 관계에 영향이 생긴다.

### 완료 기준
- 후반부 회사 운영이 연구만 잘하면 되는 문제가 아니라 거버넌스 정치가 되도록 만든다.

---

# 6. 연구 트리와 기술 서사의 심화

## 6.1 기술 트리를 단순 해금망에서 연구철학의 지도로 확장
### 현재 한계
기술 수와 카테고리 폭은 충분하나, 많은 선택이 효율적 최단 경로 탐색으로 수렴할 가능성이 있다.

### 구현 항목
- 기술 노드에 단순 수치 보너스 외에 **부작용, 철학, 경로 종속성**을 부여한다.
- 상충 기술을 일부 도입한다.
  - 오픈소스 우선 vs 독점 플랫폼화
  - 속도 우선 vs 안전 우선
  - 대규모 데이터 수집 vs 개인정보 친화적 학습
  - 범용성 우선 vs 도메인 특화
- 특정 기술 조합이 제도/이벤트를 여는 구조를 만든다.
- 연구 완료가 바로 끝이 아니라, **도입 비용**과 **조직 흡수 시간**을 추가한다.

### 완료 기준
- 기술 트리가 단순 checklist가 아니라 회사 정체성을 만드는 의사결정이어야 한다.

---

## 6.2 기술 역사 맥락을 플레이에 실질 반영
### 구현 항목
- 각 기술에 역사 맥락 문구뿐 아니라 시장 기대, 언론 반응, 경쟁사 자극 효과를 부여한다.
- 특정 시대 이전에 너무 앞선 기술을 밀면 다음 같은 현상을 만들 수 있다.
  - 투자자 이해 부족
  - 시장 부재
  - 인재 부족
  - 비용 과다
  - 규제 미비로 인한 backlash
- 반대로 너무 늦으면 경쟁사와 시장에서 매력을 잃는다.

### 완료 기준
- 기술 시점이 단순 잠금이 아니라 타이밍 전략이 되어야 한다.

---

# 7. 모델 개발, 평가, 배포, 서비스 운영의 심화

## 7.1 모델을 "한 번 만들고 끝"이 아닌 제품 라인업으로 확장
### 현재 한계
모델 설계/훈련/배포는 잘 구현되어 있으나, 모델 라인업 관리, 세대 교체, 카니발라이제이션, 유지보수의 재미가 아직 크지 않다.

### 구현 항목
- 모델 포트폴리오 시스템 추가
  - frontier flagship
  - cheap inference model
  - coding specialist
  - safety model
  - enterprise private model
  - open-source community line
- 모델 간 cannibalization과 보완 관계를 둔다.
- 동일 아키텍처의 후속 세대가 이전 모델을 대체하거나 특정 시장에 남게 만든다.
- 모델 retire, merge, distill, fork 기능을 만든다.

### 완료 기준
- 플레이어가 모델 하나만 키우는 것이 아니라 제품 포트폴리오를 관리하게 되어야 한다.

---

## 7.2 평가 체계를 시장과 규제, 안전성으로 분기
### 현재 한계
벤치마크와 상대 성능은 구현되어 있지만, 평가 결과가 곧바로 시장과 규제, 브랜드에 다층적으로 작용하는 정도는 더 늘릴 수 있다.

### 구현 항목
- 평가 축을 명확히 분리한다.
  - capability
  - reliability
  - cost efficiency
  - latency
  - safety
  - interpretability
  - misuse risk
- 특정 채널은 특정 축을 더 중시한다.
  - 정부는 안전과 통제성
  - 소비자는 속도와 체감 성능
  - 기업은 신뢰성과 통합성
  - 오픈소스는 접근성과 커뮤니티성
- 외부 평가 기관/언론/독립 벤치마크 이벤트를 도입한다.

### 완료 기준
- 모델이 성능이 높아도 안전이나 비용이 낮으면 특정 시장에만 성공해야 한다.

---

## 7.3 서비스 운영을 지속 의사결정으로 확장
### 현재 한계
서비스 채널은 이미 존재하지만, 실제 운영 중 발생하는 비용-품질-사고-고객 유지 구조가 더 복잡해질 필요가 있다.

### 구현 항목
- 서비스 운영에 다음 지표를 추가한다.
  - SLA
  - latency tier
  - hallucination complaint
  - moderation load
  - enterprise churn
  - government audit risk
  - infra saturation
- 운영 사고를 이벤트가 아니라 시스템으로 만든다.
  - 과부하
  - 지역 장애
  - 유해 출력 확산
  - 데이터 유출
  - 프롬프트 인젝션 이슈
- 운영팀/온콜/레드팀/모더레이션 인력의 필요성을 만든다.

### 완료 기준
- 배포 이후가 자동 수익 기계가 아니라 또 다른 전략 영역이 되어야 한다.

---

## 7.4 안전성과 정렬을 독립 승부축으로 강화
### 현재 한계
안전 기술과 안전 엔딩은 있으나, 안전성이 성장/정치/평판과 강하게 얽힌 구조는 더 강화할 수 있다.

### 구현 항목
- 레드팀, 안전 감사, 외부 검증, 릴리즈 게이트 도입
- 안전성 부족 상태에서 릴리즈를 강행하면 누적 리스크가 쌓이도록 한다.
- 안전 연구는 느리지만 장기 보상과 특정 시장 접근성을 높인다.
- 내부 AI 안전 감사를 서비스와 연구, 칩 설계에도 연결한다.
- 안전성 이벤트는 선택지를 넘어 지속 제약을 남기게 한다.

### 완료 기준
- 안전은 착한 플레이용 보너스가 아니라, 속도와 다른 형태의 강력한 전략이 되어야 한다.

---

# 8. 컴퓨트, 데이터, 인프라, 칩 프로그램의 통합 심화

## 8.1 GPU/클라우드/코로케이션/데이터센터 간 경제성 차별화 강화
### 현재 한계
컴퓨트 시스템은 이미 높은 밀도를 가지지만, 장기적인 계약 전략과 리전 전략, 유지보수 이벤트가 더 들어갈 수 있다.

### 구현 항목
- 클라우드 사업자별 성향 강화
  - 가격 안정성
  - 계약 할인
  - 정부 친화성
  - 리전 가용성
  - 특정 가속기 제공
- 코로케이션과 데이터센터는 다음 변수를 갖게 한다.
  - 전력 계약
  - 냉각 효율
  - 확장 대기시간
  - 지역 리스크
  - 환경 반발
- 자가 인프라는 CAPEX와 OPEX, 규제, 지역 리스크를 모두 고려하게 만든다.

### 완료 기준
- 컴퓨트 확보가 숫자 증가가 아니라 공급망 전략처럼 느껴져야 한다.

---

## 8.2 데이터 자산을 실제 전략 자원으로 격상
### 현재 한계
데이터 믹스와 적합도는 구현되어 있지만, 데이터의 법적/윤리적/지역적 성격과 장기 효과는 더 발전할 수 있다.

### 구현 항목
- 데이터 타입에 다음 메타정보를 추가한다.
  - 법적 민감도
  - 수집 난이도
  - 최신성
  - 편향 위험
  - 지역 사용 제한
  - 산업별 가치
- 데이터 확보 방식 추가
  - 구매
  - 파트너십
  - 스크래핑
  - 합성 생성
  - 사용자 로그
  - 정부/기업 계약 데이터
- 데이터 출처에 따라 이벤트와 규제가 바뀌게 한다.
- 데이터 리니지와 감사 로그를 만든다.

### 완료 기준
- 데이터가 단순한 TB가 아니라 전략적이고 위험한 자산이 되어야 한다.

---

## 8.3 내부 칩 프로그램을 조직/시장/정치와 더 강하게 연결
### 현재 한계
칩 프로그램은 이미 강한 독립 시스템이지만, 회사 전체와 국제 공급망에 미치는 파장을 더 키울 수 있다.

### 구현 항목
- 칩 프로그램이 다음과 연결되게 한다.
  - HW 인재 쟁탈전
  - 파운드리 블록 정치
  - 수출 통제
  - 전력 소비 구조
  - 서비스 수익성
  - 정부 보조금
- 프로그램 성공/실패가 한 번의 보너스가 아니라 장기 파급효과를 남기게 한다.
  - 내부 인프라 우위
  - 투자자 신뢰 상승
  - 시장 내러티브 변화
  - 공급망 의존도 변화
- 칩 개발은 세대별 철학 차이를 가지게 한다.
  - 학습 최적화
  - 추론 최적화
  - 에너지 효율
  - 정부 보안 인증

### 완료 기준
- 칩 프로그램이 미니게임이 아니라 회사 정체성과 중후반 판세를 뒤집는 전략축이 되어야 한다.

---

# 9. 경제, 자본시장, 기업가치 시스템 확장

## 9.1 재무제표와 현금흐름 설명력을 강화
### 현재 한계
수익과 비용은 존재하지만, 왜 돈이 늘고 줄었는지 한눈에 읽는 경험은 약하다.

### 구현 항목
- 월간 재무 리포트 화면 추가
  - revenue by channel
  - cost by type
  - cash flow trend
  - runway trend
  - gross margin
  - inference margin
  - R&D intensity
- 전월 대비 증감과 원인을 표시한다.
- 각 비용/매출 항목에 툴팁으로 계산식과 modifier 출처를 보여준다.

### 완료 기준
- 플레이어가 재무를 감으로가 아니라 원인 중심으로 해석할 수 있어야 한다.

---

## 9.2 기업가치 계산을 시장 심리와 미래 기대까지 반영
### 현재 한계
기업가치는 다요인 기반이지만, 투자자 심리와 업계 거품, 내러티브 변화의 반영을 더 확장할 수 있다.

### 구현 항목
- 밸류에이션을 다음 층위로 분리한다.
  - fundamentals
  - hype premium
  - safety discount
  - geopolitical discount
  - platform premium
  - hardware moat premium
- 업계 사이클 이벤트를 만든다.
  - AI 버블
  - 규제 공포
  - 오픈소스 쇼크
  - 연산비용 붕괴
  - 대형 사고 이후 신뢰 붕괴
- 같은 매출이라도 서사와 리스크에 따라 valuation이 달라지게 한다.

### 완료 기준
- 회사 가치가 단순한 점수판이 아니라 시장이 붙인 해석처럼 느껴져야 한다.

---

## 9.3 투자 라운드를 협상 시스템으로 심화
### 현재 한계
현재는 조건 충족 후 진행하는 구조가 강하다.

### 구현 항목
- 투자 라운드에 term sheet를 도입한다.
  - 청산 우선권
  - 보드 시트
  - 투자자 보호 조항
  - 성장 압박 KPI
  - 국가/산업 제한
- 여러 투자자 제안을 비교하게 한다.
- 유치 실패, down round, bridge round, structured financing을 넣는다.
- 정부 자금과 민간 자금의 trade-off를 강화한다.

### 완료 기준
- 투자 유치가 버튼 누르기보다 전략적 협상이 되어야 한다.

---

# 10. 이벤트, 서사, 선택의 후폭풍 확장

## 10.1 이벤트를 "지속 상태 생산 장치"로 전환
### 현재 한계
이벤트와 체인 구조는 많이 좋아졌지만, 더 많은 이벤트가 장기 상태를 남겨야 한다.

### 구현 항목
- 이벤트 결과로 `country policy`, `board mood`, `talent bloc`, `media narrative`, `regulatory scrutiny`, `public trust debt` 같은 상태를 남긴다.
- 이벤트 상태는 다른 이벤트의 가중치와 선택지, 잠금 조건을 바꾸게 한다.
- 이벤트는 과거 선택을 요약해 회수해야 한다.
  - “당시 당신은 군사 계약을 선택했었다”
  - “데이터 수집 논란 이후 또다시…”
- 이벤트 체인을 사람 중심, 시장 중심, 정치 중심, 기술 공포 중심으로 확대한다.

### 완료 기준
- 이벤트 하나가 끝난 뒤에도 세계가 달라진 느낌이 남아야 한다.

---

## 10.2 내부 갈등과 회사 문화 시스템 강화
### 구현 항목
- 회사 문화 축을 추가한다.
  - speed culture
  - academic culture
  - mission culture
  - secrecy culture
  - public accountability
- 문화는 채용, 이탈, 내부 갈등, PR, 안전 사고, 오픈소스 정책에 영향을 준다.
- 문화가 지나치게 한쪽으로 치우치면 반작용 이벤트가 발생한다.

### 완료 기준
- 회사가 단순한 CEO 의사결정 기계가 아니라 조직 생태계처럼 느껴져야 한다.

---

## 10.3 철학적, 공포, AGI 서사 확장
### 구현 항목
- 후반부 철학/공포 이벤트를 더 늘리되, 장식이 아니라 시스템적으로 연결한다.
- AGI 전후의 질문을 다양화한다.
  - 누구를 위해 공개할 것인가
  - 국가와 공유할 것인가
  - 안전을 위해 속도를 늦출 것인가
  - 사람의 노동, 권력, 군사 균형에 어떤 영향을 줄 것인가
- 후반부는 뉴스와 UI 분위기, 사운드, 지도 표현까지 더 불안정하고 장엄해지게 설계한다.

### 완료 기준
- 후반부가 단지 숫자가 커진 상태가 아니라 감정적으로도 특별해야 한다.

---

# 11. 승리 조건과 엔딩의 질적 확장

## 11.1 엔딩을 점수판이 아닌 서사 결산으로 강화
### 현재 한계
승리/패배 조건은 명확하지만, 엔딩별 여운과 플레이 요약이 부족하다.

### 구현 항목
- 엔딩마다 다음을 요약한다.
  - 세계에 남긴 영향
  - 실업/호감도/안전성 결과
  - 주요 인재의 최후
  - 경쟁사 운명
  - 국가 질서 변화
  - 회사의 유산
- 엔딩 후 신문/다큐 스타일 몽타주, 기업 연감, 연대표를 보여준다.
- 뉴게임+ 보너스는 단순 수치가 아니라 지난 철학을 반영한다.

### 완료 기준
- 한 판이 끝났을 때 "점수"보다 "내가 어떤 회사였는가"가 기억에 남아야 한다.

---

## 11.2 엔딩 분기 추가
### 구현 항목
- 현재 엔딩 외에 다음 계열을 고려한다.
  - 공공 인프라형 AGI
  - 군산복합체형 패권
  - 오픈소스 해방형 질서
  - 규제로 꺾였으나 윤리적 유산을 남긴 실패
  - 매각/인수합병 엔딩
  - 내부 쿠데타/CEO 실각 엔딩
  - 국가에 흡수된 프로젝트 엔딩
- 엔딩은 단순 조건 충족 외에 플레이어 과거 선택과 상태 플래그를 반영한다.

### 완료 기준
- 같은 승리 결과여도 어떤 방식으로 거기 도달했는지에 따라 다른 엔딩이 나와야 한다.

---

# 12. UI/UX, 정보 구조, 설명 가능성의 대폭 강화

## 12.1 인과관계 가시화
### 현재 한계
시스템이 많아질수록 플레이어는 원인을 놓치기 쉽다. 현재도 정보량은 꽤 많지만, "왜 그런가"를 한 눈에 보여주는 구조는 더 필요하다.

### 구현 항목
- 다음 UI를 추가한다.
  - 월간 보고서
  - 경쟁사 비교 보고서
  - 시장 변화 원인 패널
  - 규제 리스크 대시보드
  - 인재 만족도/이탈 리스크 패널
  - 모델 포트폴리오 손익표
  - 지역별 매출 지도
- 수치 옆에 영향 요인 breakdown을 넣는다.
- 기술/정책/이벤트가 어떤 수식을 바꿨는지 표시한다.

### 완료 기준
- 숙련자가 외부 메모 없이도 게임 내부 정보만으로 전략을 세울 수 있어야 한다.

---

## 12.2 정보 계층 재설계
### 구현 항목
- 각 화면을 다음 3층으로 분리한다.
  1. 상위 요약
  2. 원인/세부 분해
  3. 역사/배경/설명
- 중요 경고는 색만으로 표시하지 말고 이유와 해결 방법을 함께 보여준다.
- 탭 간 이동 시 사용자가 잊기 쉬운 정보는 고정 위젯으로 남긴다.
  - 현금
  - 런웨이
  - 가장 큰 병목
  - 가장 위험한 경쟁사
  - 가장 임박한 이벤트

### 완료 기준
- 플레이어가 필요한 정보에 도달하기 위해 불필요한 팝업 왕복을 하지 않아야 한다.

---

## 12.3 튜토리얼을 조작 설명에서 전략 교습으로 확장
### 구현 항목
- 단계형 튜토리얼 외에 **개념 튜토리얼**을 추가한다.
  - 런웨이 관리법
  - 연구-서비스 균형
  - 국가 선택의 의미
  - 배포 전략 차이
  - 경쟁사 대응법
  - 안전성과 속도의 trade-off
- 실패 리플레이용 힌트 시스템을 넣는다.
- 튜토리얼이 현재 판의 선택을 반영해 맞춤 조언을 하게 한다.

### 완료 기준
- 초심자가 게임 메커니즘의 목적을 이해하도록 해야 한다.

- 고대비 UI 테마
- 모션/글리치 효과 강도 조절 옵션

### 완료 기준

# 14. 기술 아키텍처와 유지보수성 개선

## 14.1 main.js 분해
### 현재 한계
`main.js`가 지나치게 많은 책임을 가진다. 이는 기능 추가 속도를 둔화시키고 회귀 위험을 높인다.

### 분해 목표
다음 책임을 별도 모듈로 나눈다.
- 화면 라우팅
- 모델 설계/훈련 UI
- 서비스 운영 UI
- 외교/경쟁사 패널
- 월간 보고서 생성기
- 생성 화면
- 게임오버/엔딩 렌더러
- 이벤트 모달 렌더러
- 명령 처리 계층

### 완료 기준
- `main.js`는 오케스트레이션만 담당하고, 시스템 로직과 HTML 조립을 대규모로 직접 들고 있지 않아야 한다.

---

## 14.2 전역 참조 제거와 명령형 API 정리
### 현재 한계
`window.game` 참조가 많고, UI가 로직에 직접 접근하는 패턴이 많다.

### 해야 할 일
- command dispatch 계층 도입
- UI는 가능한 한 상태 읽기와 command 호출만 하도록 정리
- 이벤트/뉴스/사운드/지도 펄스는 이벤트 버스 또는 signal layer로 통합

### 완료 기준
- 테스트에서 전체 게임 인스턴스를 띄우지 않고도 개별 시스템을 검증할 수 있어야 한다.

---

## 14.3 데이터 주도형 콘텐츠 파이프라인 확장
### 구현 항목
- 이벤트, 경쟁사 교리, 투자자 성향, 국가 정책 템플릿, 언론 기사 템플릿, 엔딩 서술을 데이터 파일로 이동
- schema validation 도입
- 콘텐츠 lint 도구 추가
  - 누락 i18n 키
  - 존재하지 않는 techId/modelId/countryId
  - 체인 참조 오류
  - dead event 탐지

### 완료 기준
- 콘텐츠 추가가 코드 수정 없이 가능한 비율을 높인다.

---

## 14.4 성능 최적화와 렌더링 비용 절감
### 현재 한계
현재 구조는 많은 화면을 `innerHTML` 중심 재렌더링하며, 규모가 커질수록 비용이 커질 수 있다.

### 구현 항목
- 실시간 갱신이 필요한 영역과 정적 영역을 분리한다.
- 리스트 가상화가 필요한 화면을 판별한다.
  - 뉴스 로그
  - 이벤트 기록
  - 인재 목록
  - 모델 포트폴리오
- 맵, 패널, 로그, 모달 업데이트를 미세 조정한다.
- FPS 및 렌더 비용 디버그 오버레이를 개발 모드에서 제공한다.

### 완료 기준
- 후반 대규모 상태에서도 UI 응답성이 무너지지 않아야 한다.

### 목표

# 19. 최종 완성 기준
AGI Manager가 높은 완성도에 도달했다는 판단은 다음 기준을 동시에 만족할 때 가능하다.

## 시스템 기준
- 초중후반이 서로 다른 질문을 던진다.
- 경쟁사가 실제 전략 세력처럼 작동한다.
- 국가와 지정학이 장기 전략을 바꾼다.
- 인재와 조직 갈등이 숫자를 넘어 기억에 남는다.
- 안전성과 규제가 진짜 승부축이다.
- 후반부가 가장 재미있다.

## UX 기준
- 플레이어가 모든 핵심 수치의 원인을 해석할 수 있다.
- 복잡한 시스템이 있어도 길을 잃지 않는다.
- 튜토리얼 없이도 UI 자체가 의도를 설명한다.

## 콘텐츠 기준
- 이벤트는 결과를 남긴다.
- 엔딩은 플레이 요약이 아니라 세계의 결산이다.
- 반복 플레이마다 다른 이야기 구조가 나온다.

## 기술 기준
- 빌드와 저장이 신뢰 가능하다.
- 테스트와 시뮬레이터로 밸런스를 검증할 수 있다.
- 콘텐츠 추가가 구조를 무너뜨리지 않는다.

# 20. AI용 실행 힌트
마지막으로, 이 문서를 읽는 AI는 아래 순서를 기본 사고 틀로 삼는 것이 좋다.

1. **블로커 정리**: 빌드, 버전, 문서, 저장, 구조 위험 제거
2. **핵심 구조 설계**: 캠페인 4막, 경쟁사 교리, 국가 정책 행위자화
3. **사람과 정치 추가**: 인재 관계망, 이사회, 투자자, 언론, 규제기관
4. **경제와 운영 심화**: 지역 시장, 서비스 운영, 안전 감사, 공급망
5. **설명 가능성 추가**: 보고서, 리스크 대시보드, 원인 로그
6. **엔딩과 유산 강화**: 결산, 뉴게임+, 세계 반영
7. **마지막으로 콘텐츠를 늘린다**: 이벤트, 시나리오, 국가, 경쟁사, 엔딩 추가

핵심은 단순하다.

**이 게임은 이미 재료가 많다. 이제 필요한 것은 더 많은 재료가 아니라, 재료끼리 서로를 바꾸는 화학이다.**
